# SQL Dialect Notes

## PostgreSQL (Primary Target)

The skill defaults to PostgreSQL. Key things to remember:

- Use `TIMESTAMPTZ` ‚Äî never bare `TIMESTAMP`
- `BIGINT GENERATED ALWAYS AS IDENTITY` for auto-increment PKs (SQL standard, PG 10+)
  - Legacy: `BIGSERIAL` (still works but not recommended)
- `TEXT` is fine for variable-length strings ‚Äî no need for `VARCHAR(255)`
- Prefer CHECK constraints on TEXT when values evolve often; use ENUM for stable value sets
- `JSONB` (not `JSON`) for JSON storage ‚Äî it's indexed and binary-stored
- Foreign keys are NOT automatically indexed ‚Äî always add them manually
- `now()` = transaction time, `clock_timestamp()` = wall clock time
- Use `ON CONFLICT DO UPDATE` for upserts (not `INSERT OR REPLACE`)
- Partial indexes with `WHERE` clause for sparse data
- `INCLUDE` in indexes for covering index / index-only scans
- Row-Level Security (RLS) for multi-tenant isolation
- Native partitioning (RANGE, LIST, HASH) since PG 10

### Identity Column Syntax
```sql
id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
-- or, if you need to sometimes insert explicit values:
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

### Flyway / Migration Scripts
- For versioned migrations, prefer fail-fast DDL to detect schema drift early
- Use broad `IF [NOT] EXISTS` only for repeatable/bootstrap scripts
- Use `CREATE INDEX CONCURRENTLY` for large tables (separate migration, outside transaction)
- Add FK constraints with `NOT VALID` then `VALIDATE` separately for zero-downtime

## MySQL / MariaDB

Key differences from PostgreSQL:

| Feature | PostgreSQL | MySQL / MariaDB |
|---------|-----------|-----------------|
| Auto-increment PK | `BIGINT GENERATED ALWAYS AS IDENTITY` | `BIGINT AUTO_INCREMENT` |
| Timestamp with TZ | `TIMESTAMPTZ` | `DATETIME` (enforce UTC at connection level) |
| Text type | `TEXT` (no limit) | `TEXT` (64KB), `MEDIUMTEXT` (16MB), `LONGTEXT` (4GB) |
| JSON | `JSONB` (binary, indexable) | `JSON` (text-based, limited indexing) |
| Enum | CHECK constraint on TEXT | `ENUM('a','b','c')` (native but same modification issues) |
| FK auto-index | No (manual) | Yes (InnoDB creates index automatically) |
| Auto-update timestamp | Trigger function | `ON UPDATE CURRENT_TIMESTAMP` (built-in) |
| Upsert | `ON CONFLICT DO UPDATE` | `ON DUPLICATE KEY UPDATE` |

```sql
-- MySQL auto-update timestamp (no trigger needed)
updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

### InnoDB Specifics
- Always use InnoDB engine (default since MySQL 5.5.5)
- Clustered index on PK ‚Äî choose PKs carefully (sequential integers preferred)
- `innodb_buffer_pool_size` should be ~70% of available RAM
- Online DDL supported for many operations (`ALGORITHM=INPLACE`)

## SQLite

Key differences:

- No native date type ‚Äî store as `TEXT` (ISO 8601: `'2024-01-15T10:30:00Z'`) or `INTEGER` (Unix epoch)
- No auto-increment sequences ‚Äî `INTEGER PRIMARY KEY` auto-increments
- Foreign key enforcement is off by default ‚Äî enable with `PRAGMA foreign_keys = ON`
- CHECK constraints are enforced
- No `ALTER TABLE ADD CONSTRAINT` ‚Äî constraints must be defined at table creation
- Weak typing: column types are advisory (any column can hold any type)
- No concurrent writers ‚Äî single-writer, multiple-reader (WAL mode helps)
- No partitioning, no stored procedures, no triggers returning values
- Great for: local dev, embedded apps, mobile. Not for multi-writer production.

```sql
-- SQLite auto-increment
id INTEGER PRIMARY KEY  -- auto-increments without explicit keyword
```

## Snowflake

Key differences for data warehouse use:

| Feature | PostgreSQL | Snowflake |
|---------|-----------|-----------|
| Timestamp with TZ | `TIMESTAMPTZ` | `TIMESTAMP_TZ` |
| Auto-increment | `GENERATED ALWAYS AS IDENTITY` | `AUTOINCREMENT` or `IDENTITY` |
| Large integers | `BIGINT` | `NUMBER(38, 0)` |
| Money | `NUMERIC(19,4)` | `NUMBER(18, 2)` |
| JSON | `JSONB` | `VARIANT` (semi-structured) |
| FK enforcement | Yes | No (metadata only) |
| Indexing | B-Tree, GIN, BRIN, GiST | Micro-partitions + clustering keys |

```sql
-- Snowflake identity
id NUMBER(38,0) IDENTITY(1,1) PRIMARY KEY
```

- No traditional indexes ‚Äî performance comes from micro-partitions and clustering keys
- Schema design focuses on star/snowflake schemas for analytics
- Foreign keys exist for documentation but are not enforced
- Use clustering keys on columns frequently used in WHERE/JOIN

## BigQuery

Key differences for analytics:

- `TIMESTAMP` is always UTC-aware (no timezone confusion)
- Supports informational PRIMARY KEY/FOREIGN KEY constraints, but they are not enforced
- No auto-increment ‚Äî generate IDs with `GENERATE_UUID()` or ROW_NUMBER()
- Partitioned tables by `DATE(created_at)` for cost and performance
- Clustered tables by frequently-filtered columns (up to 4)
- Nested/repeated fields (`ARRAY<STRUCT<...>>`) instead of junction tables
- Column names are case-insensitive
- Billing by bytes scanned ‚Äî `SELECT *` is expensive, always select specific columns

```sql
-- BigQuery partitioned and clustered table
CREATE TABLE project.dataset.order (
  order_id STRING NOT NULL,
  customer_id STRING NOT NULL,
  total_cents INT64,
  status STRING,
  created_at TIMESTAMP
)
PARTITION BY DATE(created_at)
CLUSTER BY customer_id, status;
```

## CockroachDB

Increasingly common for globally distributed applications. Wire-compatible with PostgreSQL but
with key differences:

| Feature | PostgreSQL | CockroachDB |
|---------|-----------|-------------|
| Auto-increment | `GENERATED ALWAYS AS IDENTITY` | `UUID DEFAULT gen_random_uuid()` (preferred) |
| Sequences | Fast, local | Cross-node coordination (slow) ‚Äî avoid for PKs |
| Partitioning | RANGE, LIST, HASH | Geo-partitioning by region |
| Isolation | Read Committed (default) | Serializable (only option) |
| Stored procedures | Full PL/pgSQL | Limited (no PL/pgSQL) |

```sql
-- CockroachDB preferred PK (avoids hotspots from sequential inserts)
id UUID PRIMARY KEY DEFAULT gen_random_uuid()

-- If you must use integers, use UNIQUE ROWID (hash-distributed)
id INT8 PRIMARY KEY DEFAULT unique_rowid()
```

**Key differences:**
- Prefer UUID or hash-based PKs ‚Äî sequential integers cause write hotspots on a single range
- Serializable isolation is the only option (stronger than PG's default Read Committed)
- `IMPORT` for bulk loads instead of `COPY`
- Geo-partitioning to pin data to specific regions for compliance

## General Cross-Dialect Tips

When the user doesn't specify a dialect, ask ‚Äî or default to PostgreSQL and note differences.

When migrating between dialects:
1. Audit all TIMESTAMP vs TIMESTAMPTZ handling
2. Check enum/constraint support
3. Verify auto-increment / identity syntax
4. Review index strategies (PG doesn't auto-index FKs; MySQL does)
5. Test all triggers ‚Äî syntax differs significantly
6. Check JSON support (JSONB vs JSON vs VARIANT)
7. Verify partitioning syntax and capabilities
8. Check transaction isolation defaults
